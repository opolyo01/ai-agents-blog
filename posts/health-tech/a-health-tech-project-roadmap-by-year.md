If the curriculum gives you the classes, the next question is obvious:

what should you actually build each year?

Because this path only works if it turns into proof.

Not just coursework.

Not just grades.

Proof that you can take healthcare problems, use technical tools, and turn them into working systems.

That means projects.

This is the project roadmap I would use if the goal were to graduate with a portfolio that makes sense for health-tech, clinical data, healthcare analytics, or applied AI roles.

## The rule for every year

Each year should produce at least one thing that is:

- concrete
- shippable
- explainable in an interview
- tied to a real workflow or dataset

The worst outcome is finishing school with a transcript but no evidence of applied skill.

## Year 1: learn the tools and ship simple things

The first year is not about building a startup.

It is about getting comfortable with the basics:

- Python
- SQL
- CSVs and data cleaning
- statistics
- simple dashboards

### Project 1: patient vitals dashboard

Build a small dashboard using synthetic or public health data.

Goal:

- ingest a dataset
- clean it
- calculate basic metrics
- visualize results

Possible features:

- trends over time
- heart rate and blood pressure summaries
- risk flag thresholds
- filtering by patient or group

What this proves:

- Python
- data cleaning
- charts
- basic analytics

### Project 2: appointment no-show analysis

Use a public healthcare dataset and analyze missed appointments.

Goal:

- explore predictors
- build a simple model
- explain the result clearly

What matters here is not model sophistication.

What matters is whether you can move from messy data to a usable conclusion.

### Project 3: SQL-backed health records mini app

Build a very small records system with:

- patient table
- visits table
- medications table
- queries for summaries

This does not need to be pretty.

It needs to prove you understand:

- data modeling
- relational structure
- CRUD plus reporting

### Year 1 outcome

By the end of year one, the portfolio should show:

- data cleaning
- Python
- SQL
- basic UI or dashboard work
- at least one healthcare-themed dataset

That is enough to start applying for early internships, research support roles, or data-heavy student jobs.

## Year 2: move from analysis to systems

Year two should look less like class exercises and more like applied workflow tools.

This is where the portfolio should start feeling specific.

### Project 4: clinician workflow dashboard

Build a dashboard for a realistic healthcare operations problem.

Examples:

- triage queue
- care-team workload
- appointment flow
- patient follow-up tracking

The point is to design around an operational workflow, not just a dataset.

What this proves:

- product thinking
- workflow design
- real-world framing

### Project 5: explainable healthcare prediction tool

Take a supervised learning use case, but keep it interpretable.

Examples:

- readmission risk
- appointment no-show risk
- simple disease screening support

Important constraint:

the output should be understandable.

That means:

- feature importance
- human-readable summary
- explicit caveats

This is much more valuable than an opaque model with marginally better metrics.

### Project 6: health data pipeline

Build a mini pipeline that does something like:

```text
CSV / API input
-> cleaning
-> validation
-> storage
-> dashboard or report
```

This is one of the best possible year-two projects because it starts looking like real systems work.

What this proves:

- data engineering thinking
- validation
- basic pipeline design
- reliability mindset

### Internship goal by Year 2

By the end of year two, the student should be able to apply for roles like:

- health-tech intern
- data analyst intern
- healthcare operations analyst
- clinical systems support intern
- product/data internship in digital health

That is the point where the path starts compounding.

## Year 3: add scale, systems, and applied AI

Year three is where the work should start looking less academic.

The goal is to build systems that feel closer to production patterns.

### Project 7: healthcare data platform prototype

Build a small but real platform around healthcare data.

Possible architecture:

```text
ingestion
-> database
-> API
-> dashboard
-> scheduled reporting
```

Possible features:

- auth
- role-based access
- audit logging
- exportable reports

This is the kind of project that starts to separate strong candidates from everyone else.

### Project 8: AI-assisted documentation workflow

This is a strong modern project if done carefully.

Examples:

- summarizing patient intake notes
- extracting action items from care coordination text
- generating operational reports from structured inputs

The key is not "use AI because AI is trendy."

The key is to build:

- validation
- review steps
- constraints
- transparency

That shows you understand AI as part of a workflow, not magic.

### Project 9: streaming health operations view

Build something that feels more like a live system.

Examples:

- incoming patient queue
- monitoring dashboard
- alert stream
- live operational metrics

You do not need hospital-scale infrastructure.

You do need to show you understand:

- snapshots
- streaming updates
- filtering
- operator-facing UX

This is one of the most underrated project categories because it teaches systems thinking, not just analysis.

## Year 4: specialize and polish

Year four is not the time to start from zero.

It is the time to turn the strongest ideas into something coherent.

### Project 10: capstone that combines everything

The capstone should combine:

- domain understanding
- data
- systems design
- usability
- some amount of AI or automation if appropriate

Examples:

- care coordination operations platform
- patient monitoring analytics system
- AI-assisted clinical workflow support tool
- hospital operations dashboard with predictive insights

This should be the portfolio centerpiece.

### Project 11: research or industry collaboration

If possible, this is where you tie your work to:

- a research lab
- a professor
- a hospital-adjacent group
- a health-tech company

This matters because real-world constraints improve project quality fast.

### Project 12: portfolio cleanup

Most students skip this and it hurts them.

Take the strongest two or three projects and make them easy to understand:

- clear README
- screenshots
- architecture notes
- short demo
- explanation of tradeoffs

Messy projects hidden in random repos do not help nearly as much as a clean portfolio.

## What the project stack should prove by graduation

By graduation, the portfolio should demonstrate:

- data analysis
- healthcare domain context
- SQL and data modeling
- workflow thinking
- dashboard and reporting ability
- basic ML or prediction
- applied systems design
- responsible use of AI inside a workflow

That is a much stronger signal than "I took some classes in CS and biology."

## What to avoid

Some project traps are predictable:

### 1. Generic machine learning notebooks

If the whole project is just:

- import dataset
- train model
- print accuracy

it will not differentiate you.

### 2. Overbuilding infrastructure too early

Do not spend six months pretending to be a cloud platform engineer before you can even explain the user problem.

### 3. No domain framing

A health-tech project without a clear healthcare workflow is just a generic software project with medical colors.

### 4. No narrative

Projects need a story:

- what problem
- for whom
- why current workflows are bad
- what your system improves

That narrative is what turns technical work into interview leverage.

## Internship strategy

The goal is not to wait until senior year.

The path should look more like:

- Year 1: build simple but real projects
- Year 2: use those to land a first internship or relevant role
- Year 3: build stronger systems and get a better internship
- Year 4: convert that into full-time positioning

That sequence is what turns the curriculum into actual advantage.

## Final takeaway

The curriculum matters.

But the portfolio is what makes the path real.

The strongest students in this track will not just know health systems and data in theory.

They will have already built:

- tools
- dashboards
- pipelines
- workflow systems
- AI-assisted processes

inside a healthcare context.

That is what creates leverage.
